# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

Personal dotfiles for sloria, expressed as Ansible roles. Provisions three kinds of targets from one playbook:

- **local** (macOS workstation, `localhost`)
- **remote** (mac mini server, reached over SSH)
- **orb** (Debian VMs created via OrbStack on the local Mac)

The repo is expected to be cloned to `~/dotfiles` — the wrapper scripts in `bin/` and the playbook variables (`dotfiles_home`) hard-code that path.

## Common commands

All day-to-day operations go through wrappers in `bin/` (which is added to `$PATH` once provisioned):

```sh
./bin/dot-local                  # Provision/update the local mac (all roles)
./bin/dot-local git python       # Run only the given role tags
./bin/dot-remote [tags...]       # rsync repo to mac-mini, then provision it
./bin/dot-vm                     # List OrbStack VMs (alias for `orb list`)
./bin/dot-vm create [name]       # Create + provision a Debian VM (default name: "debian")
./bin/dot-vm provision [name]    # Re-provision an existing VM (or all under [orb] if no name)
./bin/dot-vm rm <name>           # Delete VM, inventory entry, and SSH config block
./bin/upgrades                   # brew upgrade + rustup update
./bin/cleanup                    # brew/docker/uv/mise cache cleanup + ~/tmp
```

Lint (run before committing playbook changes):

```sh
mise run lint        # equivalent to `prek run --all-files`
```

`prek` (a pre-commit drop-in) runs prettier and ansible-lint per `.pre-commit-config.yaml`. Note: the ansible-lint hook's `files:` regex matches `^ansible/.*\.ya?ml$`, so it currently doesn't actually scan playbooks at the repo root — invoke `ansible-lint provision.yaml roles/` directly when you want lint coverage.

Run a single role/play manually (what `dot-local` wraps):

```sh
ansible-playbook provision.yaml -i hosts.ini -i hosts.local.ini --limit local --tags git
```

`ansible-galaxy install -r requirements.yaml` is run automatically by `dot-vm`; run it manually after editing `requirements.yaml` for the local/remote flows.

## Architecture

### Playbook structure

`provision.yaml` is the only playbook. It has four plays, each gated by host group:

1. `hosts: orb` — VM lockdown (`orb_lockdown`), `wireguard`, `geerlingguy.docker`. Runs as `become: true`.
2. `hosts: all` — the cross-platform core: `git`, `gh`, `package_manager`, `utilities`, `zsh`, `python`, `mise`, `rust`, `neovim`, `eza`, `bat`, `delta`.
3. `hosts: mac` — macOS-only: `macos`, `lctl`.
4. `hosts: local` — local-mac-only: `kubernetes`, `jrnl`, `k9s`, `ghostty`.

Every role is registered with a `tags: [<role_name>]` entry, so `--tags <role>` selects that one role across whichever play(s) include it.

### Inventory + groups

- `hosts.ini` is committed and defines `[local]`, `[remote]`, and the `[mac:children]` group combining them. `[mac]` is what selects macOS-only plays; `[local]` is the workstation; `[remote]` is the mac mini.
- `hosts.local.ini` is **generated by `dot-vm`** and is gitignored. It holds the `[orb]` group plus per-VM `ansible_host`, `git_name`, `git_email`, `dotfiles_user_home`. Both inventory files are passed to `ansible-playbook` together (`-i hosts.ini -i hosts.local.ini`).
- `dotfiles_user_home` is per-host (set in inventory) because the macOS user is `steve` locally but `stevenloria` on the mac mini and `$USER` inside VMs. Use `{{ dotfiles_user_home }}` rather than `~` or `ansible_env.HOME` in tasks.

### Group vars and local overrides

- `group_vars/all.yaml` — defaults including `git_name`, `git_email`, `utilities_packages`.
- `group_vars/local.yaml` / `group_vars/remote.yaml` — `mac_homebrew_packages`, `mac_cask_packages`, `mac_mas_packages` per host group.
- `group_vars/**/*.local.yaml` and `hosts.local.ini` — gitignored. This is the convention for machine-specific or secret-bearing config (e.g. `wireguard_profile` for an orb VM goes in `group_vars/orb.local.yaml`).
- Per-user secrets at runtime live in `~/.localrc`, which is sourced last by `roles/zsh/files/zshrc.link`. `dot-vm` writes `GH_TOKEN` there for new VMs.

### Role conventions

A role typically contains:

- `tasks/main.yaml` — install + symlink/template steps. **OS branching is done with `when:` on every install task**, using `ansible_facts["system"] == "Darwin"` for macOS and `ansible_facts["os_family"] == "Debian"` for Linux. The `community.general.homebrew` module is used on macOS; `ansible.builtin.apt` (with `become: true`) is used on Debian. Some packages (e.g. `uv`) install via upstream shell script on Debian when no apt package is available.
- `defaults/main.yaml` — role variables (e.g. `zsh_prezto: true`, `package_update`).
- `files/`, `templates/` — assets. Files ending in `.link` are intended to be symlinked into `$HOME` (see `roles/git/tasks/main.yaml` for the pattern: `ansible.builtin.file: state=link force=true`). Existing user files are usually backed up to `*.bak` first via a `failed_when: false, changed_when: false` copy.
- `*.zsh` files at the role root — **auto-loaded into the shell** by `roles/zsh/files/zshrc.link`, which globs `$ZSH/**/*.zsh`. Load order matters: anything matching `*/path.zsh` is sourced first, then prezto, then everything except `path.zsh`/`completion.zsh`/`final.zsh`, then completions, then `final.zsh`. Underscore-prefixed names like `_env.zsh`/`_path.zsh` exist so they sort early within the "everything else" pass. Putting an alias or env var in a role's `aliases.zsh` is the standard way to extend the shell without touching the zsh role itself.
- `bin/` at the role root — same idea: `$ZSH/bin` is on `$PATH`, so dropping a script in `roles/<role>/bin/` makes it available everywhere. (Currently only the top-level `bin/` is wired in; check `roles/zsh/_env.zsh` if you add one.)

### OrbStack VM specifics

- VMs are created with `orb create debian:trixie <name>`. The codebase deliberately avoids `orb create --isolated` because of upstream bug [orbstack#2409](https://github.com/orbstack/orbstack/issues/2409) (breaks SSH agent forwarding). When that's fixed, `orb_lockdown` can be deleted and `dot-vm` simplified — both files have TODO comments marking the spots.
- `orb_lockdown` substitutes for `--isolated` by: (1) running a systemd oneshot at boot that unmounts `/mnt/mac`, `/Users`, `/Applications`, `/Library`, `/Volumes`, `/private`; (2) dropping a `/etc/profile.d/` script that unsets `SSH_AUTH_SOCK`. It does **not** isolate networking — VMs still reach the host via `host.orb.internal`.
- `dot-vm create` writes an SSH config block (marked `# Added by dot-vm: <name>`) so VS Code Remote-SSH and `ssh <name>` work. `dot-vm rm` removes that block and the inventory line. The `sed -i ''` calls in `dot-vm` are macOS BSD sed syntax — the script is mac-only.
- WireGuard `.conf` profiles go in `roles/wireguard/files/profiles/` and are gitignored.

## Conventions worth following

- **YAML extension**: always `.yaml`, never `.yml` (this was an explicit cleanup — see commit `ff5a668`).
- **Idempotency**: every task must be safely re-runnable. Use `creates:` on `command:`, `force: false` on backup copies, `state: link force: true` on symlinks, and prefer module state over shell scripts.
- **No bootstrap tag**: `dot-local`'s comment mentions skipping a `bootstrap` tag, but no role currently uses it. Add it if you introduce one-time setup.
- **ansible-lint skips**: `var-naming[no-role-prefix]` and `latest[git]` are intentionally disabled in `.ansible-lint`. Don't re-enable without understanding why (group_vars use unprefixed names like `git_name`, `mac_cask_packages`).
- **Adding a new role**: create `roles/<name>/tasks/main.yaml`, register it in the appropriate play in `provision.yaml` with `tags: ["<name>"]`, and put any shell integration in `roles/<name>/aliases.zsh` (or `*.zsh`).
- **Editing macOS package lists**: `mac_homebrew_packages` / `mac_cask_packages` live in `group_vars/local.yaml` (workstation) and `group_vars/remote.yaml` (mac mini) — they're separate intentionally. Cross-platform CLI tools go in `utilities_packages` in `group_vars/all.yaml` instead.
